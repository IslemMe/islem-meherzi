The Problem Many Developers Face

GitHub Actions has become the go-to tool for automating workflows—whether you’re running tests, building reports, or deploying applications. It’s powerful, integrated, and for public repositories, completely free.

But here’s the catch:

GitHub Actions free tier only works for public repositories.

If your repository is private, you’re limited to about 2,000 minutes per month on the Free plan. Public repositories? Unlimited minutes. The same applies to GitHub Pages—deploying a website from a public repo is free, but deploying from a private repo requires upgrading your account.

So what do you do when your code contains:

  • Proprietary algorithms?
  • Client data that can’t be exposed?
  • Research methods not ready for publication?
  • API keys or sensitive configurations?

Most people think they have two choices:

  1. Make the repository public (and risk exposing everything)
  2. Pay for private repo minutes

But there’s a third way—and it’s smarter than both.

The Solution: Public Workflow, Private Code

Here’s the key insight: Your workflow configuration doesn’t need to live in the same repository as your code.

You can create a public repository that hosts only your GitHub Actions workflows, while your actual code stays locked in a private repository. The two communicate securely using Personal Access Tokens (PATs).

Why This Works

✅ Your proprietary code stays private
✅ You get unlimited free GitHub Actions minutes
✅ You can showcase your automation process
✅ Outputs (reports, dashboards) can be published without exposing source code
✅ Perfect for reproducible research where methods are confidential but results should be shared

Real-World Example: Automated Weekly Reports

Let me show you how this works with a concrete scenario.

The Challenge

A healthcare organization needed weekly automated reports. The workflow required:

  • Connecting to a secure database (private credentials)
  • Running R-based data transformations (proprietary algorithms)
  • Generating a Quarto report
  • Publishing the final HTML to a public website

The problem: All code had to remain private, yet automation was needed every Monday morning—without exceeding free GitHub limits.

The Solution

Private Repository: client-health-pipeline

  • Contains: R scripts, data transformations, credentials
  • Status: Fully private

Public Repository: automated-reporting

  • Contains: Only the .github/workflows/ folder
  • Status: Public (to access unlimited free minutes)
  • Purpose: Triggers automation, connects securely to private repo, publishes results

The Outcome

✅ Private code remains protected
✅ Reports generate and publish automatically
✅ Zero additional GitHub costs
✅ Workflow transparency maintained

This approach works for automated reports, R package testing, ETL jobs, Shiny app deployment, and reproducible research—combining security, automation, and cost efficiency.

Step-by-Step Implementation

Step 1: Create Your Private Repository

This is where your actual work lives.

# Create a new private repository on GitHub
# Name it: my-private-analysis
# Initialize with a .gitignore

What goes here:

  • All your source code
  • Data processing scripts
  • Configuration files
  • Any sensitive information

Critical: Add a .gitignore file:

# Credentials and secrets
.env
.Renviron
config.yml
*.key

# Data files
data/raw/*
*.csv
*.xlsx

# Outputs
output/*
cache/*

Step 2: Create Your Public Workflow Repository

This repository will be public and contain only automation logic.

# Create a new PUBLIC repository
# Name it: automated-workflows

What goes here:

  • .github/workflows/ folder with your YAML files
  • Optional: README explaining your automation approach
  • Optional: Sample scripts (no sensitive code)

Step 3: Generate a Personal Access Token (PAT)

Your public workflow needs permission to access your private repository.

  1. Go to GitHub Settings → Developer settings → Personal access tokens → Tokens (classic)
  2. Click “Generate new token (classic)”
  3. Name it: workflow-private-repo-access
  4. Set expiration: 90 days (for security)
  5. Select scopes: ✅ repo (full control of private repositories)
  6. Click “Generate token”
  7. Copy the token immediately (you won’t see it again!)

Step 4: Add the Token as a Secret

In your public workflow repository:

  1. Go to Settings → Secrets and variables → Actions
  2. Click “New repository secret”
  3. Name: PERSONAL_ACCESS_TOKEN
  4. Value: Paste your PAT from Step 3
  5. Click “Add secret”

Security note: Secrets are encrypted and automatically redacted from logs.

Step 5: Create Your Workflow

In your public repository, create .github/workflows/automated-report.yml:

name: Weekly Report Generation

on:
  schedule:
    - cron: '0 6 * * 1'  # Every Monday at 6 AM UTC
  workflow_dispatch:      # Allows manual trigger

jobs:
  generate-report:
    runs-on: ubuntu-latest
    
    steps:
      - name: Checkout private code
        uses: actions/checkout@v4
        with:
          repository: yourusername/my-private-analysis
          token: ${{ secrets.PERSONAL_ACCESS_TOKEN }}
          path: private-code
      
      - name: Setup R
        uses: r-lib/actions/setup-r@v2
        with:
          r-version: '4.3.0'
      
      - name: Install R packages
        run: |
          cd private-code
          Rscript -e 'install.packages(c("tidyverse", "purrr"))'
      
      - name: Run analysis
        run: |
          cd private-code
          Rscript analysis.R
      
      - name: Render Quarto report
        run: |
          cd private-code
          quarto render report.qmd
      
      - name: Upload report
        uses: actions/upload-artifact@v4
        with:
          name: weekly-report
          path: private-code/output/report.html
      
      - name: Deploy to GitHub Pages
        uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: private-code/output

Step 6: Test Your Workflow

  1. Go to the Actions tab in your public repository
  2. Select your workflow
  3. Click “Run workflow” (manual trigger)
  4. Monitor the execution logs
  5. Security check: Ensure no secrets appear in logs

Other Powerful Use Cases

Automated Testing for Private R Packages

Run R CMD check on every commit without exposing source code:

name: R Package Check
on: [push, pull_request]
jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          repository: yourname/private-r-package
          token: ${{ secrets.PERSONAL_ACCESS_TOKEN }}
      - uses: r-lib/actions/setup-r@v2
      - run: R CMD check .

Scheduled Data Pipeline Execution

Run ETL jobs daily without manual intervention:

on:
  schedule:
    - cron: '0 2 * * *'  # 2 AM daily
jobs:
  etl:
    runs-on: ubuntu-latest
    steps:
      - name: Run data pipeline
        run: python etl_pipeline.py

Reproducible Research Workflows

Generate manuscript outputs while keeping methods confidential:

- name: Render manuscript
  run: quarto render manuscript.qmd
- name: Publish results only
  uses: actions/upload-artifact@v4
  with:
    name: published-paper
    path: output/manuscript.pdf

Shiny App Deployment

Deploy Shiny apps from private code to public servers:

- name: Deploy to shinyapps.io
  env:
    SHINYAPPS_TOKEN: ${{ secrets.SHINYAPPS_TOKEN }}
  run: |
    Rscript -e 'rsconnect::deployApp()'

What You Can Publish Safely

Even with private code, you can share:

✅ Final reports (HTML, PDF)
✅ Data visualizations and dashboards
✅ Documentation about your methodology
✅ Workflow configurations (YAML files)
✅ Logs (with sensitive info redacted)

What stays private:

❌ Source code with proprietary algorithms
❌ Raw data or personal information
❌ API keys and credentials
❌ Unpublished research methods
❌ Client-specific business logic

Common Issues & Solutions

“Authentication Failed” Error

Cause: PAT doesn’t have right permissions or expired.

Solution:

  1. Check token expiration
  2. Regenerate with repo scope
  3. Update secret in repository settings

“Resource not accessible by integration”

Cause: GITHUB_TOKEN can’t access private repos.

Solution: Always use PERSONAL_ACCESS_TOKEN for private repo access, not the default GITHUB_TOKEN.

Secrets Appearing in Logs

Cause: Accidental print of sensitive variables.

Solution:

- name: Debug (safe)
  run: |
    echo "Workflow running..."
    # NEVER: echo ${{ secrets.PERSONAL_ACCESS_TOKEN }}

Wrapping Up

Automation shouldn’t require compromising security. With this public-workflow, private-code approach, you get:

✅ Unlimited free automation
✅ Complete code privacy
✅ Reproducible workflows that can be audited
✅ Professional portfolio of your automation skills

This setup saves developers from the chaos of managing complex automations or exposing private code: reports generate automatically, data stays secure, and only polished results are shared.

Whether you’re a data scientist, researcher, or developer with confidential code, this pattern lets you embrace automation without exposure.
